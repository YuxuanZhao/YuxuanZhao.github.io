HTTP vs HTTPS

HTTP协议以明文方式发送内容，不提供任何方式的数据加密，如果攻击者截取了Web浏览器和网站服务器之间的传输报文，就可以直接读懂其中的信息，因此，HTTP协议不适合传输一些敏感信息

安全套接字层超文本传输协议HTTPS，为了数据传输的安全，HTTPS在HTTP的基础上加入了SSL（Secure Sockets Layer）协议，SSL依靠证书来验证服务器的身份，并为浏览器和服务器之间的通信加密。HTTPS协议是由SSL+HTTP协议构建的可进行加密传输、身份认证的网络协议，比http协议安全。SSL的使用是公私钥。

HTTPS缺点：会使页面的加载时间延长近50%，增加10%到20%的耗电，此外，HTTPS协议还会影响缓存，增加数据开销和功耗，甚至已有安全措施也会受到影响也会因此而受到影响。而且HTTPS协议的加密范围也比较有限，在黑客攻击、拒绝服务攻击、服务器劫持等方面几乎起不到什么作用。SSL证书需要钱，功能越强大的证书费用越高。

常见状态码

  • 100：继续，请求者应当继续提出请求。 服务器返回此代码表示已收到请求的第一部分，正在等待其余部分。
  • 101：切换协议，请求者已要求服务器切换协议，服务器已确认并准备切换。
  • 200：成功
  • 301：网页已经永久移动到新位置，自动跳转到新位置
  • 302：临时移动，暂时从另一个位置响应，下次用原位置来请求就行
  • 304：对客户端有缓存情况下服务端的一种响应
  • 401：密码错误
  • 403：禁止，服务器拒绝请求（你的IP被列入黑名单/过多地访问此网站(一般是用采集程序)，被防火墙拒绝访问了/以http方式访问需要ssl连接的网址/连接的用户过多，可以稍后再试）
  • 404：服务器找不到网页
  • 500 ：服务器内部错误，服务器遇到错误，无法完成请求。
  • 501 (尚未实施) 服务器不具备完成请求的功能。 例如，服务器无法识别请求方法时可能会返回此代码。
  • 502 (错误网关) 服务器作为网关或代理，从上游服务器收到无效响应。
  • 503 (服务不可用) 服务器目前无法使用(由于超载或停机维护)。 通常，这只是暂时状态。
  • 504 (网关超时) 服务器作为网关或代理，但是没有及时从上游服务器收到请求。
  • 505 (HTTP 版本不受支持) 服务器不支持请求中所用的 HTTP 协议版本。

前端存储

Cookie

服务器发送到浏览器，保存在本地，下一次发送请求的时候会一并发送。

主要用于：user personalized，track user behavior，保存用户是否登陆的信息

特点：大小4kb，同一个域名一般只能有20个，支持过期时间，到时间会自动销毁，每次同一个域的HTTP请求都会带上此域名下的Cookie，可以设置为http only，不会被客户端的JS访问。

localStorage

不手动清除，永远都不会过期。使用键值对存储，按域名保存在数据库文件中。

5～10MB，仅保存在客户端，不与服务器通信，重启浏览器都仍然存在。

Session Storage

只存在于当前的标签页，刷新后存在，关闭后清除。与localStorage统一的API

IndexedDB

大量结构化的数据存储，几百MB甚至更多，键值对存在对象仓库，主键不能重复

五层模型协议

应用层：HTTP，FTP（对于传输的内容有了更多的定义：文件传输，电子邮件，文件服务）

传输层：TCP，UDP（稳定/实时的传输，端对端的接口）

网络层：IP（单纯的传输，什么也不能保证，为数据包选择路由）

数据链路层：SLIP，CSLIP（传输有地址的帧以及错误检测功能）

物理层：ISO2110，IEEE802（以二进制数据形式在物理媒体上传输数据）

Websocket

比传统轮询(Traditional Polling)好在不用每次都新建一个HTTP请求。不需要Axios，可以应用于在线的游戏/聊天室的server和client间的双向传输。一个应用层协议，它必须依赖 HTTP 协议进行一次握手 ，握手成功后，数据就直接从 TCP 通道传输。

数据的传输是以frame的形式进行传输的，不用考虑到数据大小导致的长度标志位不足够的情况。

使用socket.io这一个跨平台的WebSocket通信库，重点在on监听和emit发送

客户端

传输文件的具体过程

  1. 服务器端侦听某端口，接受WebSocket请求，后面可用nginx作反向代理，外部看到的将是80端口
  2. 客户端连接服务器的WebSocket地址，连接成功后，首先传送一个NEW_FILE的数据包，里面带上要传输的文件名
  3. 服务器端收到NEW_FILE包后，解析出文件名，并创建目标文件，再回复ACK_NEW_FILE的数据包
  4. 客户端收到ACK_NEW_FILE的数据包后，检查回应的code，如是成功码，则启动一个线程，该线程负责将源文件的数据封装成多个FILE_DATA数据包，传送这些FILE_DATA数据至服务器端
  5. 服务器端接收FILE_DATA数据包，解析出里面的文件数据，将文件数据写入文件
  6. 客户端发送完源文件数据后，再传送一个FILE_END数据包，该文件包中带上源文件的MD5值
  7. 服务器端收到FILE_END数据包后，比对源文件的MD5值与目标文件的MD5值，如相同，则认为传输成功，并返回ACK_FILE_END数据包，里面带上成功码
  8. 客户端收到ACK_FILE_END数据包，检查回应的code，如是成功码，则认为传输成功，否则认为传输失败

心跳和重连

在使用websocket的过程中，有时候会遇到网络断开的情况，但是在网络断开的时候服务器端并没有触发onclose的事件。这样会有：服务器会继续向客户端发送多余的链接，并且这些数据还会丢失。所以就需要一种机制来检测客户端和服务端是否处于正常的链接状态。因此就有了websocket的心跳了。还有心跳，说明还活着，没有心跳说明已经挂掉了。

1. 为什么叫心跳包呢？
它就像心跳一样每隔固定的时间发一次，来告诉服务器，我还活着。

2. 心跳机制是？
心跳机制是客户端每隔一段时间会向服务器发送一个数据包，告诉服务器自己还活着，同时服务器接收到之后就会回传一个数据包给客户端来确定服务器端也还活着，否则的话，需要重连~

心跳的原理：每隔一段固定的时间，向服务器端发送数据，如果在正常的情况下，服务器会返回给客户端，如果客户端通过onmessage事件能监听到的话，说明请求正常。如果没有收到的话，我们就websocket.close关闭连接。这件事会被onclose事件监听到，在onclose事件内调用reconnect事件重连。

重连的原理：每隔几秒重新调用createWebSocket的函数，同时用bool验证避免重复链接

TCP三次握手和四次挥手

三次握手

客户端 client 服务器 server
SYN seq=x =>
<= SYN ACK seq=y ack=x+1
ACK seq=x+1 ack=y+1 =>

四次挥手

客户端 client 服务器 server
FIN seq=u =>
<= ACK seq=v ack=u+1
<=可能的 传输数据，client想断开，server可能还没传完呢
<= FIN ACK seq=w ack=u+1
ACK seq=u+1 ack=w+1 =>

HTML的渲染

关键渲染路径（Critical Rendering Path）是指与当前用户操作有关的内容，比如说使用渐进式页面渲染来优化渲染速度（要不要拆分？以什么粒度拆分？）

在渲染页面之前

  1. DNS 查询
  2. TCP 连接
  3. HTTP 请求即响应
  4. 服务器响应
  5. 客户端渲染

客户端渲染

  1. 处理 HTML 标记并构建 DOM 树。
  2. 处理 CSS 标记并构建 CSSOM 树。
  3. 将 DOM 与 CSSOM 合并成一个渲染树。
  4. 根据渲染树来布局，以计算每个节点的几何信息。
  5. 将各个节点绘制到屏幕上。

跨域

请求跨域了，那请求成功发送了吗？其实是成功发送并接受到了服务器的res。但是浏览器拦截了res，因为浏览器认为AJAX获取跨域响应是不安全的。

JSONP

在html定义function F，然后用<script>发送请求的后面加上?callback=F，后台返回F(参数)，而这个方程和参数将会在html运行。JSONP是GET和异步的，所以使用AJAX。

CORS

在服务端设置 Access-Control-Allow-Origin 就可以开启 CORS。 该属性表示哪些域名可以访问资源，如果设置通配符则表示所有网站都可以访问资源。支持所有类型的HTTP请求，是根本的解决方案。

同源策略

是一种约定，它是浏览器最核心也最基本的安全功能，如果缺少了同源策略，浏览器很容易受到XSS、CSRF等攻击。所谓同源是指”协议（http）+域名（baidu.com）+端口（8080）”三者相同，即便两个不同的域名指向同一个ip地址，也非同源。

它限制Cookie，localStorage，IndexedDB，sessionStorage，DOM节点，AJAX请求被拦截
